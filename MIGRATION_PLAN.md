# AT32F456 驱动移植验证计划

## 核心原则 ⚠️

**【验证原则】不允许假设、猜测的结果做依据，一切要有明确依据**

- ❌ **禁止**：基于文档描述、理论推测、经验判断做决策
- ✅ **必须**：以代码实际实现、硬件手册规格、实测数据为准
- 📋 **验证顺序**：代码 > 数据手册 > 参考文档 > 经验
- 🔍 **存疑时**：实际运行测试，用示波器/逻辑分析仪验证，或查阅芯片勘误表

**示例**：
- ❌ 错误：文档说支持CAN-FD → 假设能用32字节
- ✅ 正确：查看芯片头文件 `SUPPORT_CAN_FD` 宏定义 → 确认 `CAN_DLC_BYTES_32` 存在 → 验证通过

## 一、移植项目背景

**源工程**: STM32H7 + HAL库 + CubeMX生成
**目标工程**: AT32F456 + AT32 BSP + WorkBench生成
**核心需求**: 移植FOC电机控制完整功能（不只是API替换，要保证时序、性能、行为一致）

## 二、需要明确的关键问题（头脑风暴）

### 2.1 硬件差异
| 项目 | STM32H7 | AT32F456 | 影响 |
|------|---------|----------|------|
| 主频 | 480MHz | 192MHz | **2.5倍性能差距，FOC ISR时间预算紧张** |
| Cortex核心 | M7 (Cache+TCM) | M4 (无cache) | 中断响应、内存访问差异 |
| FPU | 单精度 | 单精度 | 浮点性能：M7≈M4×2-3倍 |
| ADC | 16bit可选 | 12bit | 精度差异，需要确认是否影响电流环 |
| CAN | FDCAN | bxCAN | **协议数据帧格式可能不同**，过滤器不同 |
| 闪存 | 1-2MB | 256KB-1MB | **OTA分区布局需重新设计** |

### 2.2 待澄清问题
1. **硬件板差异**：AT32板和参考板的引脚分配是否完全一致？（已知ADC通道分配不同）
2. **测试条件**：是否有真实电机？还是只做信号级验证？
3. **CAN 测试**：是否有上位机？是FDCAN还是CAN兼容模式？
4. **OTA 需求**：Bootloader是否同步移植？还是先做App？
5. **编码器型号**：DPT磁双编码器接口确认（UART协议）
6. **性能基准**：参考工程ISR用时多少？AT32能否达到？

## 三、移植策略

### 3.1 双层适配
```
应用层 (foc_app, foc_fast, can_wly, ota_app, encoder)
    ↓ 直接调用
平台抽象层 (新增, 推荐)
    ↓ 适配
AT32 BSP (wk_adc, wk_tmr, wk_dma...)
```

**两种策略选择：**

**A. 直接改造**：把所有`HAL_xxx`改成`adc_/tmr_/usart_xxx`，简单但耦合度高
**B. 抽象层方案**：建立平台抽象层，应用层不改，只改抽象层
- 优点：参考代码改动最小，将来再换MCU也容易
- 缺点：多一层封装，可能影响性能

**推荐：混合方案** - FOC快速循环（ISR内）直接调用BSP（性能优先），应用层用抽象层（可读性优先）

### 3.2 移植阶段划分

```
阶段0: 准备 → 阶段1: 驱动业务API → 阶段2: 编码器
   ↓
阶段3: FOC核心(开环) → 阶段4: FOC闭环 → 阶段5: 应用协议
   ↓
阶段6: 性能优化 → 阶段7: 全功能验证
```

### 阶段1：基础驱动业务API（3-5天）

**目标**：补全ADC/TIM/USART的业务功能API，达到参考工程同等能力

#### 1.1 ADC业务层（1-2天）
**待移植API**：
```c
void ADC_FOC_Start(void);          // 启动FOC采样
void ADC_Regular_Start(void);       // 启动普通采样
void ADC_CalibrateOffsets(uint16_t n_samples);  // 零点校准

// 采样数据结构

    int32_t i_a_raw, i_b_raw, i_c_raw;
    uint16_t vdc, temp_motor, temp_mos;
    uint32_t timestamp;
} foc_adc_sample_t;

extern foc_adc_sample_t g_adc_sample;
```

**验证清单**：
- [ ] **静态校准**：板子静止，ADC读到的电流值应该≈零点（2048附近，3次校准后偏差<5）
- [ ] **采样时序**：示波器观察TP_TEST，PCCE中断频率严格等于10kHz
- [ ] **数据完整**：在ADC回调里printf一次（用 isr_print），看ABC三相数据是否合理
- [ ] **温度通道**：常温下读温度ADC值，对照ADC->℃转换公式
- [ ] **VDC通道**：实测电压（用万用表）与ADC读值对比

#### 1.2 TIM业务层（1天）
**待移植API**：
```c

void TIM1_PWM_Stop(void);           // 停止PWM
void TIM1_SetDuty(uint16_t a, uint16_t b, uint16_t c);  // 设置三相占空比

```

**验证清单**：
- [ ] **PWM波形**：示波器观察PA8/PA9/PA10六路波形
  - 频率10kHz✓
  - 占空比可调（0~50%）✓
  - 死区时间~830ns（80个时钟@96MHz）✓
  - 上下管互补✓
- [ ] **同步性**：三相波形同时对齐（中心对齐）
- [ ] **OCRef vs Output**：BRK输入低电平时，所有PWM输出关断

#### 1.3 USART业务层（1天）
**待移植API**：

**验证清单**：
- [ ] **DMA接收**：上位机发字符串，板子能接收并echo回去

- [ ] **printf不丢字**：连续打印1MB数据，无丢失

**阶段1出口标准**：

### 阶段2：编码器移植（2-3天）

**目标**：DPT磁双编码器（UART协议）正常读取角度

**关键差异**：
- 参考工程用UART2，AT32工程的UART通道需要重新分配
- HAL_UART_Receive_DMA → adc_dma + idle interrupt
- 中断回调机制不同（HAL是注册回调，AT32是直接在ISR里写）

**移植步骤**：
1. 配置USART用于编码器通信（确认引脚和波特率）
2. 移植 `DPT_CalcCRC8()` - 纯算法，无需修改
3. 改造 `DPT_Encoder_Init()` - 替换HAL UART API
4. 改造收发回调 - 在at32f45x_int.c的对应UART ISR里调用DPT函数
5. 移植 `DPT_GetLatestAngles_ISR()` - 这个是FOC用，性能敏感

**验证清单**：
- [ ] **触发频率**：每ms触发一次编码器读取，统计`trigger_count`
- [ ] **CRC正确率**：100次读取，CRC错误<1次
- [ ] **角度连续性**：手动转动电机轴，角度连续变化（无跳变）
- [ ] **零点定位**：固定位置读出角度稳定，重复读差异<0.1°
- [ ] **双编码器一致性**：电机端和负载端编码器差值在合理范围
- [ ] **ISR时间**：DPT_GetLatestAngles_ISR < 5us

## 五、风险点和应对

### 5.1 高风险项

**R1：性能不够（高概率）**
- 现象：FOC ISR超时，10kHz跑不下来
- 应对：提前用DWT测量预算，必要时降到5kHz或8kHz
- 兜底：用查表+定点数代替浮点

**R2：FDCAN→bxCAN协议不兼容（中概率）**
- 现象：万里扬协议用了FD扩展帧
- 应对：先看协议规范文档，确认数据长度
- 兜底：如果必须FD，建议换MCU或用CAN桥接芯片

**R3：编码器UART时序不一致（中概率）**
- 现象：CRC校验全错或部分错
- 应对：示波器抓UART波形，对比参考工程
- 兜底：调整波特率/采样点

**R4：ADC精度不够（低概率但严重）**
- 现象：电流环抖动大，控制不稳
- 应对：12bit ADC如果不够，用过采样
- 兜底：硬件改电流采样电路

### 5.2 低风险但要注意

- **中断优先级**：参考工程NVIC优先级方案要复盘
- **栈溢出**：FOC ISR里调用很多函数，注意栈深度
- **看门狗**：启用前先确认所有路径不会卡死

## 八、每日工作流程建议

### 标准开发流程

**上午**（3-4h）：
1. 查看前一天遗留问题
2. 实现当天核心任务
3. 编译通过，初步功能测试

**下午**（3-4h）：
1. 硬件调试验证
2. 用示波器/逻辑分析仪确认
3. 记录问题和数据

**晚上**（1-2h）：
1. 总结当天进度
2. 更新文档
3. 规划第二天任务

### 关键验证节点

每完成一个阶段，必须通过验证清单后才能进入下一阶段：
- ❌ **禁止**：功能没验证就堆新代码
- ✅ **必须**：逐层验证，确保底层稳定

### 团队协作

如果有团队支持：
- **并行任务**：一人做编码器，一人做FOC算法（Week 2）
- **代码评审**：关键代码互相Review
- **测试分工**：一人功能测试，一人性能测试（Week 4）
